# DXAX 진행자 노트

## 공통 운영 기준

- 매일 시작 전 전날 산출물 파일을 다시 열어 오늘 입력값으로 사용한다.
- 오전은 개념 정렬, 오후는 실습 집중, 8교시는 리뷰와 저장 규칙 점검으로 운영한다.
- 실습 안내는 `목표 -> 시간 -> 제출물 -> 평가 기준` 순서로만 설명한다.
- 기술 이슈가 발생하면 도구 기능 설명보다 과업 구조를 먼저 유지한다.

## 1주차 운영 메모

### 1일차

- 준비물: 과정 소개 자료, 계정 점검 체크리스트, 업무 흐름 작성 템플릿
- 운영 포인트: DX와 AX의 정의를 도구 비교로 흐리지 말고 업무 변화 관점으로 묶는다.
- 질문 유도: "내 업무에서 반복이 가장 많이 발생하는 구간은 어디인가요?"
- 산출물 점검: 업무 단계가 최소 5개 이상으로 나뉘어 있는지 확인한다.
- 대안 운영: 계정 설정이 지연되면 사례 분석 시간을 앞당기고 환경 구축은 4교시 후반으로 이동한다.

### 2일차

- 준비물: 업무 흐름도 초안, 비효율 진단표
- 운영 포인트: 참가자가 불편함만 말하지 않도록 시간, 빈도, 오류, 승인 지연 기준으로 말하게 한다.
- 질문 유도: "이 업무는 왜 아직도 수작업으로 남아 있나요?"
- 산출물 점검: 개선 후보가 3개 이상 도출되었는지, 근거가 숫자 또는 사례로 적혔는지 본다.
- 대안 운영: 직무가 다양한 반이라면 개인 사례 공유보다 공통 유형 분류부터 진행한다.

### 3일차

- 준비물: AI 적용 후보 매트릭스, 우선순위 판단 기준표
- 운영 포인트: 도구 선호보다 업무 효과와 데이터 준비도를 먼저 평가하게 한다.
- 질문 유도: "가장 쉽게 시작할 수 있는 과제와 가장 임팩트 큰 과제는 같은가요?"
- 산출물 점검: 우선순위 선정 근거가 효과, 난이도, 데이터, 위험 4항목으로 정리됐는지 확인한다.
- 대안 운영: 결정이 어렵다면 파일럿 과제 1개와 장기 과제 1개로 나눠 선택하게 한다.

### 4일차

- 준비물: 문제 정의서 템플릿, 사례 비교표
- 운영 포인트: 참가자가 해결책부터 쓰지 않게 막고, 현재 문제와 기대 결과를 먼저 쓰게 한다.
- 질문 유도: "이 문제가 해결되면 무엇이 측정 가능하게 달라지나요?"
- 산출물 점검: 현상, 원인, 영향, 기대 결과가 분리돼 있는지 본다.
- 대안 운영: 문제 정의가 모호하면 5W1H 질문으로 다시 구조화한다.

### 5일차

- 준비물: 요구사항 정의서, 데이터 항목 예시, 품질 체크리스트
- 운영 포인트: 데이터 수집 가능성과 보안 제한을 함께 검토한다.
- 질문 유도: "AI에 넣을 수 있는 정보와 넣으면 안 되는 정보는 무엇인가요?"
- 산출물 점검: 입력 데이터, 제약조건, 출력 형식, 검수 기준이 모두 적혔는지 확인한다.
- 대안 운영: 데이터가 없다면 가상 데이터셋으로 요구사항만 먼저 설계하게 한다.

## 2주차 운영 메모

### 6일차

- 준비물: 1주차 산출물 묶음, 프롬프트 템플릿 예시
- 운영 포인트: 프롬프트를 질문이 아니라 재현 가능한 업무 지시서로 설명한다.
- 질문 유도: "이 프롬프트를 다른 사람이 써도 같은 품질이 나올까요?"
- 산출물 점검: 역할, 입력, 절차, 출력 형식이 모두 포함됐는지 확인한다.
- 대안 운영: 초보자는 템플릿을 수정하게 하고, 숙련자는 직접 새 템플릿을 설계하게 한다.

### 7일차

- 준비물: 문서 유형별 구조 예시, 검수 체크리스트
- 운영 포인트: 초안 생성과 수정 작업을 한 번에 하지 않게 분리한다.
- 질문 유도: "이 문서는 누구를 설득하기 위한 문서인가요?"
- 산출물 점검: 초안과 개선본이 모두 남아 있는지, 수정 이유가 기록됐는지 확인한다.
- 대안 운영: 문서 작성 수준 차이가 크면 이메일과 보고서 트랙으로 나눠 실습한다.

### 8일차

- 준비물: 긴 문서 샘플, 요약 기준표, 인사이트 메모 양식
- 운영 포인트: 요약 길이보다 목적 적합성을 먼저 평가한다.
- 질문 유도: "같은 자료를 임원용과 실무자용으로 요약하면 무엇이 달라지나요?"
- 산출물 점검: 요약본과 인사이트 메모가 구분돼 있는지 확인한다.
- 대안 운영: 자료 난도가 높으면 공용 예제로 먼저 연습 후 개인 자료로 이동한다.

### 9일차

- 준비물: 샘플 데이터, 분석 질문 리스트
- 운영 포인트: 숫자 해석에서 원인 단정이 성급해지지 않도록 가설 언어를 유지한다.
- 질문 유도: "이 수치 변화의 가능한 설명은 몇 가지인가요?"
- 산출물 점검: 지표, 해석, 후속 질문이 분리돼 적혔는지 확인한다.
- 대안 운영: 데이터 공포가 큰 학습자에게는 텍스트 기반 해석 과제부터 시작하게 한다.

### 10일차

- 준비물: Excel 샘플 파일, 요약 프롬프트, 해석 보고 템플릿
- 운영 포인트: 표 정리 없이 바로 프롬프트를 쓰지 않게 한다.
- 질문 유도: "이 표를 AI가 읽기 쉽게 바꾸려면 무엇을 먼저 고쳐야 하나요?"
- 산출물 점검: 데이터 정리 전후 차이와 최종 보고 초안이 모두 남아 있는지 본다.
- 대안 운영: Excel 숙련도 차이가 크면 공통 정제본을 제공하고 해석에 집중시킨다.

## 3주차 운영 메모

### 11일차

- 준비물: 자동화 후보 목록, 시나리오 카드
- 운영 포인트: 기술 데모보다 자동화 가치와 적합성 판단에 시간을 더 쓴다.
- 질문 유도: "이 흐름에서 사람이 꼭 판단해야 하는 구간은 어디인가요?"
- 산출물 점검: Trigger, Action, 기대 효과가 한 문장으로 설명되는지 확인한다.
- 대안 운영: 실제 사례가 없으면 2주차 산출물 생성 흐름을 자동화 예제로 전환한다.

### 12일차

- 준비물: 자동화 흐름도 템플릿, 예외 처리 체크리스트
- 운영 포인트: 정상 경로만 그리지 말고 실패 시 알림과 재시도 규칙까지 쓰게 한다.
- 질문 유도: "입력값이 비어 있거나 형식이 틀리면 무엇이 일어나야 하나요?"
- 산출물 점검: 입력, 처리, 출력, 검수, 예외 처리 5구간이 모두 있는지 확인한다.
- 대안 운영: 구조가 복잡하면 수작업 승인 단계를 남긴 반자동 흐름으로 축소한다.

### 13일차

- 준비물: 협업 상황 시나리오, 팀 문서 양식
- 운영 포인트: 개인 최적화와 팀 표준화의 차이를 분명히 설명한다.
- 질문 유도: "팀에 적용하려면 어떤 규칙과 책임자가 필요할까요?"
- 산출물 점검: 협업 문서 초안에 역할, 입력자, 검토자, 공유 위치가 적혔는지 본다.
- 대안 운영: 팀 사례가 없으면 회의 준비-회의록-후속 조치 흐름으로 공통 과제를 준다.

### 14일차

- 준비물: 효율화 지표 예시, 업무 매뉴얼 템플릿
- 운영 포인트: 시간 절감뿐 아니라 재작업 감소와 품질 안정도 같이 다룬다.
- 질문 유도: "이 업무가 잘된 상태를 어떻게 측정할 수 있나요?"
- 산출물 점검: 절감 전략과 표준 문서 구조가 함께 정리됐는지 확인한다.
- 대안 운영: 정량화가 어려우면 오류율, 응답속도, 탐색시간 같은 대체 지표를 제안한다.

### 15일차

- 준비물: 변화 시나리오 양식, 저항 요인 목록
- 운영 포인트: 도구 홍보 시간이 아니라 변화관리 설계 시간으로 유지한다.
- 질문 유도: "누가 가장 먼저 저항할 가능성이 있고, 왜 그런가요?"
- 산출물 점검: 변화 대상, 기대 효과, 저항 요소, 대응 전략이 빠짐없이 있는지 본다.
- 대안 운영: 조직 사례가 없으면 개인 역할 변화 시나리오만 먼저 작성하게 한다.

## 4주차 운영 메모

### 16일차

- 준비물: 이미지 프롬프트 예시, 저작권 가이드
- 운영 포인트: 멋진 이미지보다 업무 목적 적합성을 우선 평가한다.
- 질문 유도: "이 이미지는 어떤 업무 장면에서 실제로 쓰일 수 있나요?"
- 산출물 점검: 프롬프트에 목적, 스타일, 구도, 금지 요소가 포함됐는지 확인한다.
- 대안 운영: 도구 접근이 어렵다면 프롬프트 설계와 레퍼런스 분석 중심으로 진행한다.

### 17일차

- 준비물: 영상 프롬프트 구조 예시, 윤리 체크리스트
- 운영 포인트: 영상 결과의 기술 품질보다 메시지 흐름과 윤리 기준을 먼저 본다.
- 질문 유도: "이 영상은 누구에게 어떤 행동을 유도하나요?"
- 산출물 점검: 장면 구성, 길이, 카메라 워킹, 활용 목적이 명시됐는지 확인한다.
- 대안 운영: 생성 시간이 길면 스토리보드와 프롬프트 설계까지를 최소 완료 기준으로 둔다.

### 18일차

- 준비물: n8n 실습 환경, 샘플 JSON, 변수 매핑 예시
- 운영 포인트: 인터페이스 설명은 짧게 하고 노드 연결 원리를 반복 확인한다.
- 질문 유도: "이 노드의 입력값과 출력값은 무엇인가요?"
- 산출물 점검: 최소 1개 워크플로우가 실행되고 로그 캡처 또는 결과 기록이 남았는지 본다.
- 대안 운영: 환경 이슈가 생기면 흐름 설계도와 수기 테스트로 대체한다.

### 19일차

- 준비물: AI 노드 예제, 디버깅 체크리스트
- 운영 포인트: 오류를 실패로 보지 말고 원인 분리 훈련으로 다룬다.
- 질문 유도: "이 자동화가 틀렸다면 어느 단계에서 처음 어긋났나요?"
- 산출물 점검: 자동화 흐름, 디버깅 메모, 개선 아이디어가 함께 제출되는지 확인한다.
- 대안 운영: 구현 난도가 높으면 고객 문의 자동응대와 뉴스 요약 중 하나만 완성하게 한다.

### 20일차

- 준비물: 1주차 문제 정의서, 2주차 프롬프트 세트, 3주차 자동화 설계안, 4주차 구현 결과
- 운영 포인트: 회고만 길어지지 않게 하고 실행 계획 작성 시간을 충분히 확보한다.
- 질문 유도: "수료 후 30일 안에 가장 먼저 실행할 한 가지는 무엇인가요?"
- 산출물 점검: 우선 과제, 일정, 협업 대상, 위험 대응, 성공 지표가 로드맵에 모두 있는지 확인한다.
- 대안 운영: 19일차와 20일차 사이 공백을 고려해 시작 30분은 산출물 복구와 재정리에 사용한다.
