# DXAX 발표자 스크립트

## 공통 사용 원칙

- 오프닝은 "왜 오늘 이 내용을 배우는가"를 먼저 설명한다.
- 전개 파트에서는 전날 산출물을 오늘 입력값으로 다시 호출한다.
- 마무리는 반드시 다음 날 과제와 연결한다.
- 실습 전환 멘트는 길게 설명하지 말고 산출물 기준과 시간만 다시 확인한다.

## 1주차: 이해와 진단

### 1일차

- 오늘의 메시지: DX와 AX를 구분하고, 내 업무를 바꿔야 할 이유를 스스로 말할 수 있어야 한다.
- 오프닝: "이 과정은 도구 사용법을 외우는 수업이 아니라, 업무를 다시 설계하는 수업입니다. 오늘은 전체 4주 구조를 먼저 잡고, 각자 업무를 어떤 관점으로 진단할지 출발점을 만들겠습니다."
- 핵심 설명 포인트:
  - DX는 프로세스를 바꾸는 관점이고 AX는 판단과 실행 방식을 바꾸는 관점이다.
  - 생성형 AI는 정답 기계가 아니라 업무 가설을 빠르게 검증하는 도구다.
  - 오늘 작성하는 업무 흐름 초안은 2일차 병목 분석의 재료가 된다.
- 실습 전환 멘트: "지금부터는 설명을 듣는 입장에서 벗어나 본인 업무를 단계로 쪼개 보겠습니다. 완벽하게 쓰는 것이 아니라, 흐름이 보이도록 쓰는 것이 목표입니다."
- 마무리: "내일은 오늘 그린 업무 흐름에서 어디가 느리고 반복적인지 찾아냅니다. 오늘 산출물은 수정하지 말고 버전 그대로 저장해 주세요."

### 2일차

- 오늘의 메시지: 업무를 잘게 나누고 비효율을 수치화해야 개선 우선순위가 선다.
- 오프닝: "어제는 업무를 펼쳐 놓았고, 오늘은 그 안의 병목을 찾습니다. 감으로 불편하다고 느낀 지점이 아니라, 시간과 반복 기준으로 말할 수 있어야 합니다."
- 핵심 설명 포인트:
  - 병목은 시간이 오래 걸리는 지점만이 아니라 승인, 탐색, 재작업이 반복되는 지점이다.
  - DX 관점의 개선 포인트와 AX 관점의 적용 후보는 겹치기도 하고 다르기도 하다.
  - 오늘의 후보 업무 목록은 3일차 우선순위 판단의 기준표가 된다.
- 브리지 멘트: "좋은 자동화는 멋진 기술에서 나오지 않고, 가장 반복적이고 가장 명확한 업무에서 시작합니다."
- 마무리: "내일은 오늘 찾은 후보 중 어떤 업무를 AI로 먼저 바꾸면 효과가 큰지 선별합니다."

### 3일차

- 오늘의 메시지: AI를 어디에 적용할지 정하는 순간부터 과정의 실전성이 결정된다.
- 오프닝: "모든 업무에 AI를 붙일 수는 없습니다. 오늘은 적용 가능성, 효과, 위험을 함께 보면서 우선순위를 정하겠습니다."
- 핵심 설명 포인트:
  - 효과가 커도 입력 데이터가 없으면 실행이 어렵다.
  - 자동화 적합성과 AI 지원 적합성은 구분해서 봐야 한다.
  - 우선순위 매트릭스는 4일차 문제 정의 문서의 근거가 된다.
- 실습 전환 멘트: "지금부터는 아이디어를 늘리는 시간이 아니라, 버릴 것을 정하는 시간입니다."
- 마무리: "내일은 가장 가치가 높은 한두 개 과제를 골라 문제 정의서 형태로 구체화합니다."

### 4일차

- 오늘의 메시지: 문제를 선명하게 정의해야 이후 프롬프트와 자동화가 흔들리지 않는다.
- 오프닝: "AI 활용이 실패하는 가장 큰 이유는 도구가 아니라 문제 정의가 흐리기 때문입니다. 오늘은 해결하고 싶은 상황을 문서로 못 박겠습니다."
- 핵심 설명 포인트:
  - 문제 정의서는 현상, 원인, 기대 결과, 제약조건을 분리해 써야 한다.
  - 좋은 문제 정의는 바로 다음 단계의 입력값 설계로 이어진다.
  - 오늘 문서는 5일차 요구사항 정의서의 기준점이다.
- 브리지 멘트: "무엇을 해결할지 명확해졌다면, 이제 AI에게 무엇을 넣고 무엇을 받아야 하는지 정리할 차례입니다."
- 마무리: "내일은 문제 정의를 데이터와 요구사항의 언어로 번역하겠습니다."

### 5일차

- 오늘의 메시지: 입력 데이터 구조를 모르면 좋은 결과물도 재현할 수 없다.
- 오프닝: "이번 주를 마무리하면서, 이제 우리는 막연한 아이디어가 아니라 문제 정의와 데이터 요구사항까지 확보하려고 합니다."
- 핵심 설명 포인트:
  - 데이터 품질은 AI 성능과 직접 연결된다.
  - 입력값, 맥락, 출력 형식을 분리해 정의해야 재사용 가능하다.
  - 오늘의 요구사항 정의서는 2주차 프롬프트 설계의 출발점이다.
- 마무리: "다음 주부터는 오늘 만든 요구사항을 실제 프롬프트와 결과물로 바꿉니다. 이번 주 산출물 4종은 하나의 묶음으로 저장해 두세요."

## 2주차: 결과물 생산

### 6일차

- 오늘의 메시지: 프롬프트는 질문이 아니라 업무 지시서다.
- 오프닝: "2주차부터는 AI를 실제로 움직입니다. 지난주에 만든 문제 정의와 요구사항을 프롬프트 구조로 변환해 보겠습니다."
- 핵심 설명 포인트:
  - 역할, 맥락, 절차, 출력 형식이 프롬프트의 기본 골격이다.
  - 같은 업무도 좋은 검수 기준이 있어야 반복 품질이 나온다.
  - 오늘 만든 템플릿은 7일차 문서 작성의 엔진이 된다.
- 마무리: "내일은 이 프롬프트를 문서 초안 생산으로 확장하겠습니다."

### 7일차

- 오늘의 메시지: 생성은 시작일 뿐이고, 완성도는 구조와 검수에서 나온다.
- 오프닝: "오늘은 이메일, 보고서, 제안서처럼 실제 업무에서 바로 써야 하는 문서를 만들어 봅니다."
- 핵심 설명 포인트:
  - 문서마다 논리 구조가 다르므로 프롬프트도 문서 유형별로 달라져야 한다.
  - 초안 생성과 후반 검수는 분리된 단계다.
  - 오늘 결과물은 8일차 요약과 재구성의 입력 자료가 된다.
- 브리지 멘트: "문서를 잘 쓰는 것과 핵심을 잘 압축하는 것은 다른 역량입니다."
- 마무리: "내일은 긴 자료를 요약하고 인사이트 메모로 바꾸는 훈련을 하겠습니다."

### 8일차

- 오늘의 메시지: 요약은 줄이기가 아니라 목적에 맞게 재배치하는 일이다.
- 오프닝: "정보가 많을수록 중요한 것은 다 읽는 능력이 아니라, 필요한 기준으로 다시 정리하는 능력입니다."
- 핵심 설명 포인트:
  - 대상, 목적, 분량이 정해져야 좋은 요약이 나온다.
  - 요약 결과는 다시 의사결정용 메모로 전환되어야 가치가 생긴다.
  - 오늘 만든 인사이트 메모는 9일차 데이터 해석의 말하기 기준이 된다.
- 마무리: "내일은 숫자와 표를 해석하는 관점으로 넘어갑니다."

### 9일차

- 오늘의 메시지: 데이터 분석의 핵심은 계산보다 해석과 질문이다.
- 오프닝: "오늘은 데이터 리터러시를 다룹니다. 숫자를 읽는 방법을 알면 AI 결과를 더 정확하게 검수할 수 있습니다."
- 핵심 설명 포인트:
  - 데이터 유형과 분석 목적을 구분해야 적절한 질문이 나온다.
  - AI 분석 도구는 패턴을 제시하지만, 맥락 해석은 사람이 책임진다.
  - 오늘 인사이트 메모는 10일차 Excel 실습 보고의 뼈대가 된다.
- 마무리: "내일은 Excel과 AI를 결합해 정리와 해석을 하나의 흐름으로 연결하겠습니다."

### 10일차

- 오늘의 메시지: 2주차의 목표는 결과물을 잘 만드는 사람에서 재현 가능한 결과를 만드는 사람으로 이동하는 것이다.
- 오프닝: "이번 주 마지막 날입니다. 지금까지의 문서 작성, 요약, 데이터 해석을 한 번에 묶어 보겠습니다."
- 핵심 설명 포인트:
  - 표 구조가 정리되어야 AI 요약의 품질이 안정된다.
  - 해석 보고는 지표, 의미, 조치 제안의 순서로 말해야 한다.
  - 오늘 정리한 프롬프트 세트와 보고 초안은 3주차 자동화 설계의 재료가 된다.
- 마무리: "다음 주에는 사람이 직접 하던 생산 흐름을 자동화와 협업 구조로 바꿔 보겠습니다."

## 3주차: 자동화와 협업

### 11일차

- 오늘의 메시지: 자동화는 반복 업무를 기계적으로 줄이는 것이 아니라, 흐름을 설계하는 일이다.
- 오프닝: "3주차는 개인 생산성을 조직 실행으로 연결하는 주간입니다. 오늘은 자동화 후보를 다시 선별하고 시나리오 초안을 설계하겠습니다."
- 핵심 설명 포인트:
  - 규칙이 분명한 업무부터 자동화 후보가 된다.
  - Trigger와 Action을 이해하면 업무가 흐름으로 보이기 시작한다.
  - 오늘 초안은 12일차 구조 설계에서 세부 로직으로 발전한다.
- 마무리: "내일은 예외 처리까지 포함한 설계안으로 한 단계 더 구체화합니다."

### 12일차

- 오늘의 메시지: 자동화 설계의 완성도는 정상 흐름보다 예외 처리에서 드러난다.
- 오프닝: "자동화가 실제 업무에서 쓰이려면 실패했을 때 어떻게 대응할지까지 설계해야 합니다."
- 핵심 설명 포인트:
  - 입력, 처리, 출력, 검수, 예외 처리의 다섯 블록으로 자동화를 설명한다.
  - API와 Webhook은 기술 용어가 아니라 연결 방식의 차이다.
  - 오늘 흐름도는 13일차 팀 협업 적용의 기본 구조다.
- 브리지 멘트: "개인 흐름이 명확해지면, 이제 팀 맥락에서 무엇이 달라지는지 봐야 합니다."
- 마무리: "내일은 같은 자동화 논리를 협업 환경으로 확장하겠습니다."

### 13일차

- 오늘의 메시지: AI는 개인 도구일 때보다 팀 운영 규칙 안에 들어갈 때 더 큰 효과를 낸다.
- 오프닝: "오늘은 팀 문서, 회의, 커뮤니케이션에서 AI가 어떤 역할을 맡을 수 있는지 점검합니다."
- 핵심 설명 포인트:
  - 팀 적용은 품질보다 표준화와 공유 규칙이 먼저다.
  - 협업 도구와 AI의 결합은 속도보다 정렬 비용을 줄이는 데 의미가 있다.
  - 오늘 협업 시나리오는 14일차 효율화 체계의 근거가 된다.
- 마무리: "내일은 팀 단위 효율화 기준과 문서 체계를 정리하겠습니다."

### 14일차

- 오늘의 메시지: 효율화는 더 빨리 하는 것이 아니라 덜 흔들리게 하는 것이다.
- 오프닝: "오늘은 반복 작업 감소 전략과 문서 체계화를 통해 업무 품질을 안정화하는 방법을 다룹니다."
- 핵심 설명 포인트:
  - 시간 절감만이 아니라 재작업 감소, 정보 탐색 감소도 효율화 지표다.
  - 업무 매뉴얼은 사람을 통제하는 문서가 아니라 품질을 재현하는 장치다.
  - 오늘 전략 문서는 15일차 변화 시나리오 작성에 직접 연결된다.
- 마무리: "내일은 기술 적용의 결과를 개인과 조직의 변화 언어로 정리합니다."

### 15일차

- 오늘의 메시지: AX 전환의 핵심은 도구 도입이 아니라 역할과 의사결정 방식의 변화다.
- 오프닝: "이번 주 마지막 날은 기술보다 사람과 조직을 다룹니다. 변화가 왜 필요한지, 무엇이 저항을 만드는지 정리해 보겠습니다."
- 핵심 설명 포인트:
  - 개인 변화와 조직 변화는 속도가 다르므로 분리해 설계해야 한다.
  - 변화 시나리오에는 기대 효과와 저항 관리가 함께 들어가야 한다.
  - 오늘 문서는 4주차 실행 로드맵의 변화관리 파트가 된다.
- 마무리: "다음 주에는 멀티모달 도구와 n8n 구현을 거쳐 최종 실행계획까지 마무리합니다."

## 4주차: 확장과 정착

### 16일차

- 오늘의 메시지: 이미지 생성형 AI는 표현을 바꾸는 도구가 아니라 업무 전달력을 넓히는 도구다.
- 오프닝: "4주차부터는 텍스트를 넘어 이미지와 영상, 그리고 실제 구현 도구까지 확장합니다."
- 핵심 설명 포인트:
  - 이미지 프롬프트는 목적, 스타일, 구도, 금지 요소를 함께 적어야 한다.
  - 실무 활용은 화려함보다 일관성과 사용 목적이 중요하다.
  - 오늘 산출물은 17일차 영상 기획의 비주얼 기준이 된다.
- 마무리: "내일은 움직임과 서사를 가진 영상 프롬프트로 확장합니다."

### 17일차

- 오늘의 메시지: 영상 생성은 장면을 만드는 것이 아니라 전달 흐름을 설계하는 일이다.
